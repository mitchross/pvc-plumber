# Migration: pvc-plumber v1 → v2

> Audience: anyone running `ghcr.io/mitchross/pvc-plumber:1.x` (most likely
> `1.7.0`) who is about to bump to `2.0.0`.
> If you are deploying to a fresh cluster, jump straight to
> [Option A](#option-a--fresh-cluster).
>
> **Already on v2.x and looking at v3.0.0?** The v2 → v3 cutover is much
> smaller in scope: drop the `mutate-batch-v1-job` webhook, switch the
> Kopia repo from filesystem to S3, no RBAC changes, no new cluster
> dependencies. The full migration guide lives in
> [`CHANGELOG.md`](CHANGELOG.md) under `[3.0.0] § Migration guide`. Quirks
> #1, #2, and #3 in [§ 5 below](#5-known-v2-quirks-fixed-in-v3) are
> resolved in v3.0.0 (the `1password` / `rustfs` / `kopia_password`
> hardcodes became env-var-overridable defaults).

---

## TL;DR

`v2` turns pvc-plumber into a Kubernetes **operator** — controller-runtime
manager, three admission webhooks, leader election, ServiceAccount + RBAC,
cert-manager dependency, External Secrets Operator dependency. The HTTP
`/exists` service that `v1` shipped is **preserved** behind the
`OPERATOR_MODE` feature flag, so you can deploy the v2 image as a drop-in
v1 replacement first, validate, then opt in to the operator surface.

The wire format of `/exists` is unchanged. The Kopia client is the same
code. Your existing Kyverno `apiCall` rules will still work when
`OPERATOR_MODE` is unset. The breaking changes are in the deployment
surface, not in the service interface.

---

## 1. Why this is breaking

| Concern | v1.7.0 | v2.0.0 |
|---|---|---|
| What it is | Stateless HTTP oracle | Operator + HTTP oracle (one binary) |
| K8s API access | None | ClusterRole (PVCs, ES, RS, RD, leases, events) |
| Admission webhooks | None | Three: mutate-pvc, validate-pvc, mutate-job |
| Failure mode if pod is down | `/exists` 503 → Kyverno denies | **Cluster-wide PVC create denial** for backup-labeled PVCs (failurePolicy: Fail) |
| TLS for webhooks | n/a | Required via cert-manager |
| Per-PVC kopia secret material | Generated by Kyverno policies in the consuming cluster | Generated by the operator's reconciler via ExternalSecret |
| Orphan cleanup | Out-of-band CronJob in the consuming cluster | Built into reconciler |
| Replicas | 1 (typical) | 2 with leader election (recommended) |

The takeaway: the operator can deadlock the cluster if its prerequisites
aren't in place. That's why this is a major version bump and that's why
this guide exists.

---

## 2. What you'll need before bumping

### 2.1 cert-manager

Required for the webhook serving certificate. Any working `cert-manager`
install satisfies this — the operator does not depend on a specific
version. You'll create a `Certificate` whose `secretName` is mounted
into the operator pod at `/tmp/k8s-webhook-server/serving-certs`, and the
two webhook configurations get the CA bundle injected via the
`cert-manager.io/inject-ca-from` annotation.

### 2.2 External Secrets Operator (ESO)

Required for per-PVC `ExternalSecret` rendering. Any ESO version that
supports `apiVersion: external-secrets.io/v1` works.

You also need:

- A `ClusterSecretStore` named **`1password`**. The name is hardcoded in
  `2.0.0` (see [§ 5 known quirks](#5-known-v2-quirks-fixed-in-v3)). The
  underlying provider doesn't matter — 1Password Connect, HashiCorp Vault,
  AWS Secrets Manager, the `kubernetes` provider pointing at a
  conventional `Secret`, anything that ESO supports. Only the
  `ClusterSecretStore` name is fixed.
- The store must surface a property/key path **`rustfs` → `kopia_password`**
  exactly. Also hardcoded in `2.0.0`.

If your existing `1.x` setup already uses ESO with a `ClusterSecretStore`
named `1password` and a `rustfs/kopia_password` key (it does, if you are
on the canonical reference deployment), nothing changes here.

### 2.3 ServiceAccount + ClusterRole + ClusterRoleBinding

Hand-managed in your GitOps repo. The minimal rule set is:

| API group | Resources | Verbs |
|---|---|---|
| (core) | `persistentvolumeclaims` | get, list, watch |
| `external-secrets.io` | `externalsecrets` | get, list, watch, create, update, patch, delete |
| `volsync.backube` | `replicationsources`, `replicationdestinations` | get, list, watch, create, update, patch, delete |
| `coordination.k8s.io` | `leases` | get, list, watch, create, update, patch, delete |
| (core) | `events` | create, patch |

Skipping `leases` breaks leader election (the manager fails to start).
Skipping `events` breaks controller-runtime event emission (the manager
fails to start). The operator *will not* recover gracefully from missing
RBAC — it expects to be granted everything it asks for at startup. The
hand-written `ClusterRole` is the source of truth — the codebase
intentionally does **not** carry `kubebuilder:rbac` markers, to prevent
silent regeneration drift.

### 2.4 cert-manager `Issuer` for the webhook serving certificate

A namespace-scoped `Issuer` (or `ClusterIssuer`) the webhook `Certificate`
references. A self-signed issuer is fine for an internal webhook —
admission webhooks talk to the operator service over the cluster-internal
network and the CA bundle is injected into the webhook configurations
automatically by cert-manager.

### 2.5 Namespace exclusion list — review this carefully

The operator ships a 9-entry default for `SYSTEM_NAMESPACES`:

```
kube-system, volsync-system, kyverno, argocd, longhorn-system,
snapshot-controller, cert-manager, external-secrets, 1passwordconnect
```

The same list **must** appear in the `namespaceSelector.matchExpressions`
`NotIn` clauses of your `MutatingWebhookConfiguration` and
`ValidatingWebhookConfiguration`. If you have site-local namespaces that
host controllers which create PVCs at startup (a different secret-store
backend's namespace, say), append them via the additive `SYSTEM_NAMESPACES`
env var **and** add them to both webhook configurations. The env var is
intentionally additive — it cannot drop the 9 defaults.

## 3. The migration path

Three options, from least disruptive to most disruptive.

### Option A — fresh cluster

If you're standing up a new cluster, deploy `2.0.0` from scratch with all
the prerequisites in [§ 2](#2-what-youll-need-before-bumping) in place.
Skip `1.x` entirely. The breaking-changes story doesn't apply to you.

### Option B — existing v1 user, zero-downtime cutover (recommended)

Three commits / four ArgoCD sync passes, no PVC-creation downtime.

1. **Stage 1 — drop-in image bump.** Deploy the `2.0.0` image with
   `OPERATOR_MODE=false` (or unset). The operator binary serves the legacy
   HTTP `/exists` surface only — same wire format, same env vars, same
   port. Your Kyverno `apiCall` rules continue to call it. Verify
   `/healthz`, `/readyz`, and a known-good `/exists/{ns}/{pvc}` path
   return the expected JSON. **No behavioural change yet.**
2. **Stage 2 — flip the feature flag.** Set `OPERATOR_MODE=true`. The
   reconciler activates and starts owning the
   `ExternalSecret`/`ReplicationSource`/`ReplicationDestination` triplet
   for backup-labeled PVCs. The Kyverno `generate` policies in the
   consuming cluster will *also* try to maintain those resources — both
   sides do `Get-or-Create` with identical specs, so they coexist
   harmlessly. Watch the operator logs and the affected namespaces for
   spurious creates/updates.
3. **Stage 3 — register the webhooks.** Apply the
   `MutatingWebhookConfiguration` and `ValidatingWebhookConfiguration`.
   This is the moment cluster-wide PVC creation starts going through the
   operator's admission gate. Make sure your `namespaceSelector.NotIn`
   list includes everything in [§ 2.5](#25-namespace-exclusion-list--review-this-carefully)
   *before* you apply the configurations. Also double-check the
   `cert-manager.io/inject-ca-from` annotation has populated the
   `caBundle` field on both objects (`kubectl get
   validatingwebhookconfiguration pvc-plumber -o yaml` should show a
   non-empty `caBundle`).
4. **Stage 4 — remove the consuming cluster's Kyverno + CronJob.** Delete
   the `volsync-pvc-backup-restore` and `volsync-nfs-inject` ClusterPolicy
   objects, and the `orphan-reaper` CronJob. The operator now owns
   everything they did. The
   [`talos-argocd-proxmox` `refactor-replace-kyverno` PR](
   https://github.com/mitchross/talos-argocd-proxmox/pulls?q=refactor-replace-kyverno)
   is the worked example for this stage.

If anything goes wrong at any stage, see [§ 4 Rollback](#4-rollback).

### Option C — existing v1 user, scheduled downtime

Cluster maintenance window, drain workloads, deploy `2.0.0` with all the
prerequisites in one pass, re-apply workloads. Faster to plan, riskier
because there's no observation window between stages.

## 4. Rollback

`2.0.0` rollback is fast and preserves all backups.

1. **Disable the operator surface.** `kubectl delete
   mutatingwebhookconfiguration pvc-plumber` and `kubectl delete
   validatingwebhookconfiguration pvc-plumber`. PVC creation immediately
   stops going through the admission gate; backup-labeled PVCs admit
   without `dataSourceRef` injection from this point on.
2. **Flip the feature flag.** Set `OPERATOR_MODE=false` and roll the
   Deployment. The reconciler stops watching PVCs; existing
   `ExternalSecret`/`ReplicationSource`/`ReplicationDestination` objects
   stay in place (they're not owner-referenced) and continue running
   their backup schedules.
3. **Optional — pin to `1.7.0`.** If you want to be fully off the v2
   image, change the Deployment image to `ghcr.io/mitchross/pvc-plumber:1.7.0`.
   `1.7.0` is binary-compatible with the v2 HTTP-only surface — same env
   vars, same `/exists` wire format. Re-instate the consuming cluster's
   Kyverno policies and `orphan-reaper` CronJob.

Backups are untouched throughout. The reconciler creates k8s resources;
it never reaches into the Kopia repository itself. Snapshots taken
under `2.x` are readable by `1.x` and vice versa — same Kopia repository,
same snapshot sources.

## 5. Known v2 quirks (fixed in v3)

The following are hardcoded in `2.0.0` and on the v3 followup list. None
of them block the cutover; they constrain *how* you set up the
prerequisites.

| # | Quirk | Where | v3 plan |
|---|---|---|---|
| 1 | `ClusterSecretStore` name is **`1password`** regardless of the actual provider | `internal/controller/pvc_controller.go::ensureExternalSecret` | Configurable via env var, fallback `1password` |
| 2 | `Secret` key is **`rustfs`** | same | Configurable via env var |
| 3 | Property is **`kopia_password`** | same | Configurable via env var |
| 4 | `SYSTEM_NAMESPACES` default list is hardcoded; drift between operator default and webhook `namespaceSelector` is checked at code-review time, not at runtime | `cmd/operator/main.go::defaultSystemNamespaces` and `webhooks.yaml` | Generate the webhook configurations from the same source list at deploy time |
| 5 | `volsync-system` is hardcoded as the namespace the webhook service lives in | `webhooks.yaml` (consumer-side) | Helmify / kustomize the deployment |
| 6 | `pvc-plumber.mitchross.github.io` is the leader-election lock name | `cmd/operator/main.go` | Make configurable for multi-tenant clusters |

If any of these block your deployment, file an issue. If quirk #1 / #2 /
#3 are not satisfiable on your stack (e.g. your secret store doesn't
support a key named `rustfs`), the recommended workaround for `2.0.0` is
to provision an additional `ClusterSecretStore` named `1password` that
backs onto your real store with a remap rule. v3 will let you point at
your real store directly.

## 6. Verification checklist

Before declaring the bump complete:

- [ ] Operator pod is `Running` with two replicas; one shows `acquired
      lease pvc-plumber.mitchross.github.io` in its logs.
- [ ] `kubectl get certificate -n volsync-system pvc-plumber-webhook-tls`
      shows `READY: True`.
- [ ] Both webhook configurations exist with non-empty `.webhooks[].clientConfig.caBundle`.
- [ ] `kubectl get mutatingwebhookconfiguration pvc-plumber -o
      jsonpath='{.webhooks[*].namespaceSelector}'` includes all 9 system
      namespaces in the `NotIn` list (plus any site-local additions).
- [ ] An existing backup-labeled PVC's
      `ExternalSecret`/`ReplicationSource`/`ReplicationDestination` show
      `app.kubernetes.io/managed-by: pvc-plumber` and
      `volsync.backup/pvc: <pvcname>` labels.
- [ ] A test PVC create in a non-system namespace with `backup: hourly`
      either gets a correct `dataSourceRef` injected (if a backup exists)
      or admits cleanly without one (if no backup) — and the operator
      logs the decision.
- [ ] Deleting that test PVC reaps the three child resources within one
      reconcile cycle.
- [ ] `/exists/{ns}/{pvc}` HTTP endpoint still returns the expected JSON
      for an existing path (regression check on the v1 contract).
- [ ] The consuming cluster's Kyverno `volsync-pvc-backup-restore` and
      `volsync-nfs-inject` ClusterPolicies are deleted, and the
      `orphan-reaper` CronJob is deleted.

If all of those pass, the migration is complete.
